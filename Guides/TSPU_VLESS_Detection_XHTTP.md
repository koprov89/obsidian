# Как ТСПУ ловит VLESS в 2026 и почему XHTTP — следующий шаг

**Автор:** Aleksandr (cyberscoper)
**Дата публикации:** 12 марта 2026
**Источник:** https://habr.com/ru/articles/1009542/

## Введение

Волны блокировок VLESS+Reality серверов участились в 2025 году. Разбор четырёхуровневой архитектуры детекции ТСПУ и объяснение, почему XHTTP становится следующим этапом эволюции обхода блокировок.

## Архитектура системы детекции ТСПУ

### Слой 1: Сигнатурный анализ

Первые 16–32 байта пакета сравниваются с известными паттернами. Shadowsocks и OpenVPN выявляются здесь. VLESS с корректным TLS проходит этап.

### Слой 2: TLS-Fingerprinting (JA3/JA4)

**JA3** — это хэш параметров TLS ClientHello:

```
JA3 = MD5(SSLVersion, Ciphers, Extensions, EllipticCurves, EllipticCurvePointFormats)
```

Chrome 120 имеет уникальный отпечаток. Go-библиотека `crypto/tls` генерирует иной хэш, известный системам детекции. **JA4** (от FoxIO, 2023) устойчивее к рандомизации и становится стандартом DPI-систем.

### Слой 3: Активное зондирование

Система отправляет тестовые запросы (HTTP, HTTPS, разные версии TLS). Легитимный сервис отвечает нормально, вредоносный — нет или отвечает нетипично.

### Слой 4: Поведенческий анализ

"Анализируется не содержимое пакетов, а их поведение во времени" — самый дорогой вычислительный слой. Reality уязвима именно здесь.

## TLS Fingerprinting: почему это необходимо

Без `fingerprint: "chrome"` Xray использует стандартную Go TLS-библиотеку с отличающимся набором cipher suites и порядком расширений. С параметром — используется uTLS, копирующая настоящий браузерный ClientHello.

## Reality: возможности и ограничения

Reality завершает TLS-сессию на реальном сервере (Apple/Microsoft), встраивая туннель в установленное соединение. Методы детекции, описанные в защищённой в СПбПУ (2025) работе:

1. **GREASE-анализ** — браузер вставляет мусорные расширения; паттерн имитации может отличаться
2. **IP/ASN несоответствие** — сертификат iCloud, но IP датского хостера выдаёт прокси
3. **Поведение после рукопожатия** — реальный iCloud генерирует специфичные API-запросы, VLESS-туннель — нет

## XHTTP: новая архитектура

XHTTP — транспортный протокол поверх HTTP/2 или HTTP/3. В отличие от WebSocket (постоянный двунаправленный поток), XHTTP разделяет восходящий и нисходящий трафик на отдельные HTTP-транзакции.

![[xhttp_modes.jpeg]]

### Режимы XHTTP

| Режим | Описание |
|-------|---------|
| `packet-up` | Много коротких запросов клиент→сервер, одно соединение сервер→клиент. Работает через CDN. |
| `stream-up` | Одно долгоживущее соединение в каждую сторону. Быстрее, но не через все CDN. |
| `stream-one` | Единое соединение обоих направлений. Совместимо с XTLS-Vision. |
| `auto` | Клиент выбирает сам. |

**xPaddingBytes** добавляет случайный паддинг, нормализуя размеры пакетов.

```json
{

    "port": 443,
    "protocol": "vless",
    "settings": {

        "extra": {
          "xPaddingBytes": "100-1000"

      }
    }
  }]
}
```

## Выбор SNI-донора

Используйте реальный сервис с высоким трафиком в том же регионе:

- **github.com** — Microsoft Azure, без российских PoP с 2022 года
- **www.twitch.tv** — AWS, нет российских PoP
- **www.microsoft.com** — глобальный CDN

Apple.com — плохой выбор: Apple держит IP в собственных ASN, несоответствие видно сразу.

## Проверка работоспособности

**JA3-проверка:** Подключитесь через прокси на https://scrapfly.io/web-scraping-tools/ja3-fingerprint. Отпечаток должен совпадать с обычным Chrome.

**Активное зондирование:**

```bash
curl -v https://your.domain.com \
  --resolve your.domain.com:443:<YOUR_IP>
```

Должен получиться нормальный HTTP-ответ, не TLS-ошибка.

## Сравнение подходов в 2026 году

| Параметр | VLESS + Reality | VLESS + XHTTP | VLESS + XHTTP + Reality |
|----------|-----------------|---------------|------------------------|
| TLS-отпечаток | Идеальный | Хороший (uTLS) | Идеальный |
| Активное зондирование | Выдерживает | Выдерживает | Выдерживает |
| Поведенческий анализ | Уязвим при нагрузке | Устойчив | Устойчив |
| Работа через CDN | Нет | Да (packet-up) | Частично |
| Совместимость с Vision | Да | Только stream-one | Только stream-one |
| Сложность настройки | Средняя | Средняя | Выше средней |
| Устойчивость 2026 | Снижается | Растёт | Лучшая из доступных |

## Ключевые выводы

Reality и XHTTP решают разные проблемы на разных слоях: первая защищает рукопожатие, вторая маскирует поведение после. Combined использование — оптимальное решение на текущий момент.

> **Важно:** Версии Xray на клиенте и сервере должны совпадать. Несовпадение вызывает глюки или полную неработоспособность.
